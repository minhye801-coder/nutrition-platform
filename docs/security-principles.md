# 보안 원칙

nutrition-platform은 여러 학교의 학생 개인정보를 다루는 멀티테넌트 서비스입니다. 아래 원칙은 설계와 구현 전 단계에서 지켜야 할 최소 기준입니다.

## 1. 학생 개인정보는 플랫폼 운영자 계정에 저장하지 않음
- 학생 이름, 신체/건강 정보, 상담 내용 등 개인정보는 플랫폼 운영자(개발/관리 주체)의 Google 계정이나 플랫폼 자체 DB에 저장하지 않는다.
- 모든 개인정보는 각 학교(테넌트) 소유의 Google 계정 리소스에만 저장한다.

## 2. 학교별 Google Drive/Sheets에 저장
- 데이터 저장소는 학교별로 분리된 Google Sheets/Drive를 사용한다.
- 플랫폼 서버(Cloudflare Functions)는 데이터를 중계할 뿐, 영속 저장소로 사용하지 않는다(캐시/세션 등 최소한의 예외는 별도 검토).

## 3. Gemini API Key를 GitHub나 브라우저 localStorage에 저장하지 않음
- 각 학교(영양교사)가 등록하는 Gemini API Key는 소스 코드, 커밋, 저장소(GitHub)에 절대 포함하지 않는다.
- 브라우저 `localStorage`/`sessionStorage`/쿠키 등 클라이언트 저장소에도 평문으로 저장하지 않는다.
- API Key는 서버 측 안전한 저장소(예: 암호화된 환경변수, Cloudflare의 안전한 바인딩/시크릿 저장소 등)에만 보관하고, 사용 시점에만 서버에서 호출한다.

## 4. Spreadsheet ID와 Folder ID를 공개 URL에 직접 노출하지 않음
- Google Sheets/Drive의 실제 ID는 내부 식별자로만 사용하고, 클라이언트에 전달되는 URL, 쿼리 파라미터, 공개 페이지 HTML/JS에 직접 노출하지 않는다.
- 프론트엔드/공개 링크는 실제 리소스 ID 대신 별도의 공개 식별자를 사용한다.

## 5. 공개 페이지는 schoolPublicId 또는 token 기반으로 연결
- 상담신청(`/intake`), 보호자동의(`/consent`) 등 비로그인 공개 페이지는 실제 내부 ID가 아닌 `schoolPublicId` 또는 발급된 임시 token으로 학교/케이스를 식별한다.
- token은 목적(신청, 동의 등)과 유효기간을 제한하여 발급하고, 서버 측에서 매핑 테이블을 통해 실제 리소스로 변환한다.

## 6. 민감정보는 환경변수 또는 안전한 서버 저장소 사용
- API Key, OAuth 클라이언트 시크릿, 서비스 계정 자격증명 등 모든 민감정보는 환경변수 또는 Cloudflare의 안전한 시크릿 관리 기능을 통해서만 관리한다.
- 저장소(Git)에는 `.env` 파일, 키 파일, 실제 자격증명 값을 커밋하지 않으며, `.gitignore`로 명시적으로 제외한다.
- 로컬 개발/Codespaces 환경에서도 민감정보는 코드에 하드코딩하지 않는다.

---

이 원칙들은 이후 모든 단계(특히 2단계 Google 로그인, 3단계 시트/폴더 자동 생성, 8단계 Gemini API 설정)의 설계 기준이 되며, 구현 중 원칙과 상충하는 요구가 발생하면 원칙을 우선한다.
