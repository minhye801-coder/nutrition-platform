# 아키텍처

## 1. 전체 시스템 구성

nutrition-platform은 하나의 공용 사이트에서 여러 학교의 영양교사가 각자의 Google 계정으로 로그인하여, 각자의 Google Sheets/Drive를 데이터 저장소로 사용하는 멀티테넌트 구조입니다. 플랫폼 자체는 학생 개인정보를 직접 저장하지 않고, 학교별 Google 계정 소유의 저장소에 데이터를 위임합니다.

```
                        ┌─────────────────────────┐
                        │      사용자 브라우저       │
                        │  (영양교사 / 학생·보호자) │
                        └────────────┬─────────────┘
                                     │ HTTPS
                        ┌────────────▼─────────────┐
                        │   Cloudflare Pages        │
                        │   (정적 프론트엔드, 모듈별 경로) │
                        └────────────┬─────────────┘
                                     │
                        ┌────────────▼─────────────┐
                        │  Cloudflare Functions      │
                        │  (API 라우팅, 세션/토큰 검증) │
                        └───┬─────────────────┬─────┘
                            │                 │
              ┌─────────────▼───┐   ┌─────────▼─────────────┐
              │  Google OAuth    │   │  Gemini API (사용자별) │
              │  (영양교사 로그인) │   │  (교사가 등록한 Key 사용)│
              └─────────────┬───┘   └────────────────────────┘
                            │
              ┌─────────────▼─────────────────────┐
              │   학교별 Google Sheets / Drive       │
              │   (테넌트 소유, 플랫폼 미보관)         │
              └─────────────┬─────────────────────┘
                            │
              ┌─────────────▼─────────────────────┐
              │   Google Apps Script (백엔드 보조)    │
              │   문서/PDF 생성, Drive 저장, 기존 연동  │
              └────────────────────────────────────┘
```

## 2. 구성 요소별 역할

### Cloudflare Pages
- 모듈별 경로(`/`, `/app`, `/explore`, `/intake`, `/consent`, `/setup`)를 하나의 사이트에서 서빙하는 프론트엔드 호스팅.
- 정적 자산 배포 및 CDN 역할.

### Cloudflare Functions
- 브라우저와 Google API/Apps Script 사이의 서버리스 API 레이어.
- 세션 검증, OAuth 콜백 처리, 학교(tenant)별 요청 라우팅을 담당.
- Gemini API Key, Spreadsheet ID 등 민감정보가 브라우저에 직접 노출되지 않도록 중계.

### Google OAuth
- 영양교사 개인 인증 수단. 플랫폼 자체 계정 시스템을 두지 않고 Google 계정으로 로그인.
- 로그인한 계정 권한으로 본인 소유의 Sheets/Drive에 접근.

### Google Sheets
- 학교별 데이터 저장소(학생 정보, 상담 기록, 동의 내역 등).
- 테넌트(학교)마다 별도의 스프레드시트를 사용하여 데이터 격리.

### Google Drive
- 학교별 문서, PDF, 첨부파일 저장소.
- 상담 결과지, 보호자동의서 등 산출물 보관.

### Google Apps Script
- 초기 단계에서는 Google Docs/PDF 생성, Drive 저장, 기존 시스템 기능 연동을 담당하는 보조 백엔드로 유지.
- 향후 Cloudflare Functions로 점진적 이관을 검토.

## 3. 모듈 간 연결 구조

- **관리자 설정(`/setup`)**: 학교(tenant) 등록, Google 계정 연결, Sheets/Drive 자동 생성, Gemini API Key 등록을 담당. 다른 모든 모듈이 동작하기 위한 전제 조건(tenant 초기화)을 만든다.
- **AI 영양상담 매니저(`/app`)**: 영양교사가 로그인 후 사용하는 메인 상담/관리 도구. 관리자 설정에서 생성된 tenant의 Sheets/Drive를 데이터 소스로 사용.
- **맛마을 탐험소(`/explore`)**: 학생 대상 콘텐츠/활동 모듈. 상담 매니저와 동일한 tenant 데이터를 참조하되, 공개 접근이 필요한 경우 `schoolPublicId` 기반 경로로 노출.
- **상담신청(`/intake`)**: 학생/보호자가 작성하는 공개 신청 폼. 제출 데이터는 해당 학교의 Sheets에 기록되고, 이후 관리자 모듈에서 확인.
- **보호자동의(`/consent`)**: 공개 동의 폼. 상담신청과 유사하게 token 기반 접근으로 학교를 식별하고, 동의 결과를 학교 Sheets/Drive에 기록.

상담신청과 보호자동의는 비로그인 상태로 접근하는 공개 페이지이므로, 내부 Spreadsheet ID/Folder ID를 직접 노출하지 않고 `schoolPublicId` 또는 발급된 token으로 간접 연결한다(자세한 내용은 `security-principles.md` 참고).
