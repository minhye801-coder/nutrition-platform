# 상담업무 흐름 설계 v1 (Milestone 2 재정렬)

> **문서 상태**: 설계 확정(v1). 아직 코드 구현 없음 — 이 문서는 Milestone 2A 이후 구현의 기준 문서다.
> **관련 문서**: `docs/database-schema.md`(탭/헤더 SSOT는 여전히 `functions/_lib/installTemplate.ts`), `docs/feature-priority-v1.md`(마일스톤 상세), `docs/route-and-menu-plan.md`(메뉴/라우트 상세).
> **근거**: `legacy/counseling-manager/code.gs.txt`(6892줄), `legacy/intake-consent/code.gs.txt` 재분석 결과(직전 대화 턴의 레거시 흐름 재분석 보고 참고).

## 0. 확정 원칙

1. **주 흐름은 상담접수 중심**: 공개 QR 상담신청 → 접수 검토 → 보호자동의 → 검사 → 상담케이스. 이 흐름이 케이스 생성의 기본 경로다.
2. **학생 직접 등록은 보조 흐름으로 유지**: 기능(백엔드 API, soft-delete, 중복검사)은 그대로 두되, 화면상 위치와 안내 문구만 "보조"로 재조정한다. (6절 참고)
3. **legacy 기능을 누락하지 않는다**: legacy에 있는 기능은 v1에서 축소·단순화하더라도 최소 "TBD로 명시적으로 보류"하고, 조용히 빠뜨리지 않는다. (8절 체크리스트로 검증)
4. 관계 연결은 언제나 `studentUuid → caseId`(→ 이번 범위에 새로 추가되는 `intakeId`, `consentId`) 순서로만 한다 — 이름/학년/반 매칭 금지 원칙(`docs/database-schema.md` 1절)은 상담접수 흐름에도 동일 적용.

## 1. 사용자별 흐름

이 플랫폼은 현재 "학교당 단일 로그인 계정(=영양교사=설치자)" 모델이다. 아래 4개 흐름 중 "영양교사"와 "관리자"는 같은 계정이 맡을 수 있지만, **행위의 성격이 다르므로 흐름은 구분해서 설계**한다(향후 다중 사용자/역할 분리가 필요해지면 이 구분이 그대로 역할 경계가 된다 — 10절 미결정 목록에는 없지만 설계 시 염두에 둘 전제로 기록).

### 1.1 학생/보호자 (신청자)
1. 학교가 안내한 QR코드 또는 URL(`/intake/:schoolPublicId`)로 접속 — 로그인 불필요.
2. 상담신청 폼 작성(신청자유형, 학생 정보, 상담주제, 신청내용, 연락처, 개인정보동의) 후 제출.
3. 제출 완료 화면만 확인 — 이후 진행 상황은 별도로 조회하지 않는다(조회 기능은 이번 범위 밖, 필요 시 후속 검토).

### 1.2 영양교사 (핵심 운영자)
1. 대시보드에서 신규 접수 알림 확인.
2. 상담접수 목록에서 검토 → 승인 또는 반려.
3. 승인 시 자동 생성된 상담케이스 확인, 필요하면 보호자동의 링크 발송.
4. 보호자 제출 확인 후 "교사 확인" 처리.
5. 검사 실시(업로드 또는 실시 안내) → 결과 확인.
6. 상담회기 기록, 목표 설정/점검.
7. 종결 처리.
8. (보조) 접수 없이 학생을 직접 등록하거나, 기존 재학생에게 케이스를 직접 생성해 상담을 시작할 수도 있다.

### 1.3 보호자 (동의자)
1. 문자/안내로 받은 동의 링크(`/consent/:token`)로 접속 — 로그인 불필요.
2. 케이스 요약(학생명, 상담 사유 등 최소 정보)과 동의 항목 확인.
3. 필수 동의 항목 체크 + 서명(또는 이름 확인) 후 제출.
4. 제출 완료 확인 — 이후 진행 상황은 조회하지 않는다.

### 1.4 관리자 (설치/설정 관리자)
1. 최초 로그인/설치(Google OAuth, Drive/Sheets 프로비저닝) — 기존 Milestone 1 그대로.
2. 설정 화면에서 학교명/담당자명 등 기본 정보 관리 — 기존 Milestone 1 그대로.
3. (이번 범위 밖) 향후 다중 사용자 지원 시 계정/권한 관리가 이 흐름에 추가될 수 있음.

## 3. 보조 흐름

### 3.1 교사 직접 학생 등록
- 이미 구현됨(`/students`, `functions/api/students`). 접수 없이 전입생 사전등록 등에 사용.
- 흐름/데이터 구조 변경 없음 — 화면 위치·안내 문구만 보조로 재조정(6절).

### 3.2 교사 직접 상담 시작
- 재학 중인(활성) 학생을 대상으로, 접수 절차 없이 상담케이스를 바로 생성.
- 상담케이스 메뉴에서 학생 검색(이름 검색 후 `studentUuid` 선택) → 케이스 생성.
- **legacy에는 이 경로 자체가 없다** — 완전한 v1 신규 편의 기능. "기존 학생에게 케이스 직접 생성"(3.4)과 실질적으로 동일한 액션이며, 화면상 진입점만 다를 수 있다(예: 학생관리 화면에서 바로 vs 상담케이스 화면에서).
- 이 경로로 생성된 케이스는 `intakeId`가 없다(접수를 거치지 않았으므로) — 상담케이스 헤더 설계 시 `intakeId`를 선택(nullable) 필드로 둬야 한다.

### 3.3 동의 또는 검사 생략 가능 조건
- 교사 직접 시작 케이스는 동의/검사 단계를 생략하고 바로 `상담진행`으로 이동할 수 있어야 하는 경우가 실무상 존재할 수 있다(예: 이미 오프라인으로 구두 동의를 받은 경우).
- **생략을 허용할지, 허용한다면 어떤 조건에서 허용할지는 10절 미결정 사항**으로 남긴다 — 임의로 정하지 않는다(개인정보/동의 관련 규정 이슈이므로 사용자 확인 필요).

### 3.4 기존 학생에게 케이스 직접 생성
- 3.2와 동일한 액션. 학생관리 화면(학생 상세)에서 "새 상담케이스 생성" 버튼으로 진입하는 경로로 문서화한다 — 상담케이스 메뉴의 "직접 등록"과 같은 백엔드 API를 공유.

## 4. 상태값 설계

### 4.1 상담접수 (`상담접수.status`)

| 값 | 의미 | 진입 조건 |
|---|---|---|
| `신규` | 제출 직후 기본값 | 공개 제출 시 |
| `검토중` | 교사가 검토를 시작 | 교사가 접수 상세를 열람/검토 시작 |
| `승인` | 승인 완료, 케이스 자동 생성됨 | 교사의 승인 액션 |
| `반려` | 반려 처리 | 교사의 반려 액션 |

- **legacy 대응**: legacy `처리상태`는 `신규 접수/접수 완료/검토 대기/케이스 생성` 4단계였고, 명시적 반려 상태가 없었다. v1은 이를 `신규/검토중/승인/반려`로 재정리하고 `반려`를 신규 추가한다(반려 시 처리 방식은 후속 결정 — 10절).

### 4.2 보호자동의 (`보호자동의.status`)

| 값 | 의미 | 진입 조건 |
|---|---|---|
| `미발송` | 케이스 생성 시 기본값(레코드는 있으나 링크 미발급) | 케이스 자동 생성(5절 규칙 3) |
| `발송` | 동의 링크 발급 및 발송 처리 | 교사의 "링크 발송" 액션 |
| `제출` | 보호자가 필수 항목 체크 후 제출 | 보호자 제출 |
| `확인완료` | 교사가 제출 내용 재검증 완료 | 교사의 "확인" 액션, 상담케이스를 `검사대기`로 전이시킴 |
| `거부` | 보호자가 동의를 거부 | 보호자의 명시적 거부 액션(신규) |

- **legacy 대응**: legacy `확인상태`는 `(공백)/동의 요청/교사 확인 필요/동의 완료` 4단계였고, `거부` 상태가 없었다. v1은 `미발송/발송/제출/확인완료`로 이름을 맞추고 `거부`를 신규 추가한다. 거부 시 케이스 처리 방식은 10절 미결정.

### 4.3 상담케이스 (`상담케이스.status`, 구 `currentStage`)

| 값 | 의미 | 진입 조건 |
|---|---|---|
| `동의대기` | 케이스 생성 직후 기본값 | 접수 승인(자동) 또는 교사 직접 시작(3.2) |
| `검사대기` | 보호자동의 확인 완료 | 보호자동의 = `확인완료` |
| `검사완료` | 검사 결과 저장 및 확인 | 검사 결과 저장/확인 |
| `상담예정` | 상담 일정 확정 | 검사완료 후 교사가 다음 일정 지정 |
| `상담진행` | 상담회기 진행 중 | 첫 상담회기 기록 |
| `추적관리` | 목표 점검/추적상담 진행 중 | 다음 일정이 있는 회기 기록 |
| `종결` | 상담 종료 | 교사의 종결 액션, `closedAt` 기록 |

- **legacy 대응 및 단순화 결정**: legacy `현재단계`는 8단계(`동의 대기/진단 대기/결과 확인/상담 예정/실천 중/추적상담 예정/종결 검토/종결`)였다. v1은 사용자 지정에 따라 7단계로 정리한다 — `진단 대기→검사대기`, `결과 확인→검사완료`, `실천 중→상담진행`, `추적상담 예정→추적관리`로 이름을 맞추고, **`종결 검토`는 별도 상태로 두지 않고 `종결` 처리 직전 확인 절차(화면상 확인창)로 흡수**한다(8절에서 "누락 아님"으로 명시 검증).
- **`보류` 상태는 v1에도 없다** — legacy에도 없었고(직전 분석에서 확인), 이번 사용자 지정 상태값에도 포함되지 않았다. 직전 턴에 설계했던 `진행중/보류/종결` 모델은 폐기된 것으로 최종 확정한다.

## 5. 자동 생성 규칙 (접수 승인 시)

1. **학생정보 생성 또는 연결**: `studentUuid` 매칭 우선 시도(매칭 기준은 10절 미결정) → 매칭 실패 시 신규 학생 생성(기존 `createStudent` 로직 재사용, `enrollmentStatus='재학'`).
2. **상담케이스 생성**: `caseId` 신규 발급, `studentUuid`·`intakeId` 연결, `status='동의대기'`, `openedAt=now`.
3. **보호자동의 기본 레코드 생성**: `consentId` 신규 발급, `caseId` 연결, `status='미발송'`, `consentToken`은 이 시점엔 발급하지 않고 "링크 발송" 액션 시점에 생성(legacy와 동일한 타이밍).
4. **ID 연결**: `상담접수.intakeId → 상담케이스.intakeId`(신규 FK 컬럼 필요, 4.3/8절 참고), `상담케이스.caseId → 보호자동의.caseId`(기존 스키마에 이미 존재).
5. **중복 생성 방지**: 승인 처리는 멱등이어야 한다 — 같은 `intakeId`로 이미 `상담케이스`가 존재하면 새로 만들지 않고 기존 케이스로 안내한다. 승인 API는 (a) 상담접수.status가 이미 `승인`이면 거부, (b) `상담케이스`에서 해당 `intakeId`를 가진 행이 이미 있는지 재확인, 두 겹으로 막는다.

## 6. 기존 학생관리 기능 위치

- **유지**: 백엔드 API(`functions/api/students/**`), soft-delete(`enrollmentStatus`), 중복검사, 헤더 자가치유 — 전부 변경 없음.
- **보조 기능으로 분류**: 화면 성격을 "직접 등록"에서 "접수를 거치지 않는 예외 경로"로 재정의.
- **주 대시보드 중심에서 제외**: 대시보드 첫 화면·메뉴 첫 순서는 상담접수가 차지하고, 학생관리는 메뉴 하단(설정 바로 앞)으로 이동(`docs/route-and-menu-plan.md` 1절).
- **향후 접수 승인 시 학생정보와 연결**: Milestone 2B에서 5절의 "학생정보 생성 또는 연결" 규칙이 기존 `createStudent`/`findPotentialDuplicate` 로직을 그대로 재사용하도록 구현한다 — 학생 데이터 모델을 다시 만들지 않는다.

## 7. 미결정 사항

1. 접수 승인 전 학생 중복 매칭 기준(이름+학년+반+번호 완전일치만? 유사도 매칭 허용?).
2. 보호자동의 토큰 만료 기간.
3. 검사(진단) 홈페이지 연결 방식 — 외부 검사 도구로 리다이렉트할지, 결과만 업로드받을지.
4. PDF 업로드 방식 — 교사가 직접 업로드 vs 이메일/외부 연동 자동 수집.
5. 동의 거부 시 처리 — 케이스를 자동으로 어떤 상태로 보낼지, 재발송을 허용할지.
6. 교사 직접 시작 시 동의/검사 생략 조건 — 어떤 경우에 허용할지, 감사 로그를 남길지.

### 참고: 추가로 확인이 필요해 보이는 항목 (사용자 목록 외, 설계 중 발견)

- `상담케이스` 헤더에 `intakeId`(nullable FK) 추가가 필요함 — 현재 `docs/database-schema.md` 2.5절/`installTemplate.ts`에는 없음. Milestone 2B 착수 시 헤더 확정과 함께 처리.
- "검토중" 전이 시점 — 교사가 목록에서 "열람"만 해도 자동 전이할지, 별도 "검토 시작" 버튼이 있어야 할지. 사소하지만 UX 확정 시 결정 필요.
