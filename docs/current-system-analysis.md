# 기존 시스템 분석 (예정)

> **현재 상태: 이 저장소에는 아직 기존 AI 영양상담 매니저, 맛마을 탐험소, 상담신청/보호자동의 등 기존 시스템의 실제 코드가 들어와 있지 않습니다.** 이 문서는 향후 기존 코드가 이 저장소(예: `google-apps-script/`, 또는 별도 분석용 디렉터리)에 추가되었을 때 무엇을 분석해야 하는지 미리 정리한 체크리스트입니다.

## 1. 분석 대상 시스템

- AI 영양상담 매니저
- 맛마을 탐험소
- 상담신청 페이지
- 보호자동의 페이지

각 시스템에 대해 아래 항목을 동일한 기준으로 분석한다.

## 2. 하드코딩된 값 탐지 계획

기존 코드가 특정 학교/운영자 환경에 종속되어 작성되었을 가능성이 높으므로, 다음 항목을 코드 전수 검색으로 찾아 목록화한다.

- [ ] 하드코딩된 **Spreadsheet ID** (예: `SpreadsheetApp.openById("...")` 등)
- [ ] 하드코딩된 **Folder ID** (예: `DriveApp.getFolderById("...")` 등)
- [ ] 하드코딩된 **Web App URL** (Apps Script 배포 URL, 외부 호출 엔드포인트)
- [ ] 하드코딩된 **Gemini API Key** 또는 기타 API Key
- [ ] 하드코딩된 **학교명/학교 식별 정보** (조건문, 파일명, 시트 탭 이름 등에 포함된 특정 학교 이름)
- [ ] 기타 특정 사용자/계정에 종속된 값 (이메일 주소, 사용자 ID 등)

각 항목은 발견된 파일 경로, 라인, 값의 종류를 표로 정리하고, `security-principles.md` 원칙에 따라 환경변수/설정값으로 치환할 계획을 함께 수립한다.

## 3. Apps Script 함수 분석 계획

- [ ] 전체 함수 목록 및 각 함수의 역할(트리거 함수 / 유틸 함수 / 외부 호출용 `doGet`·`doPost` 등) 정리
- [ ] 함수별 입력(파라미터, 시트 범위)과 출력(반환값, 시트 기록, Drive 파일) 정리
- [ ] 트리거 기반 실행(시간 기반, 설치형 트리거 등) 여부 확인
- [ ] 재사용 가능한 로직과 특정 학교 전용 로직 구분

## 4. 시트 입출력 흐름 분석 계획

- [ ] 각 기능이 읽고 쓰는 시트/탭 이름과 컬럼 구조 정리
- [ ] `database-schema.md`에 정의된 식별자(`tenantId`, `studentId`, `caseId`, `intakeId`, `consentId`, `assessmentId`)와의 매핑 관계 파악
- [ ] 시트 간 참조(수식, 스크립트 조회)로 발생하는 암묵적 의존 관계 정리

## 5. Drive/PDF 생성 흐름 분석 계획

- [ ] PDF/문서 생성에 사용되는 템플릿(Google Docs 템플릿 등) 목록화
- [ ] 생성된 문서의 저장 위치(폴더 구조) 및 명명 규칙 정리
- [ ] 생성 트리거(사용자 액션 vs 자동 트리거) 파악
- [ ] 생성된 문서에 대한 접근 권한 설정 방식 확인 (공유 범위, 링크 공개 여부 등)

## 6. 분석 결과의 활용

이 분석은 `development-roadmap.md`의 4~7단계(각 모듈 이전)에서 실제 이관 작업의 입력 자료로 사용된다. 분석이 끝나면 이 문서를 실제 조사 결과로 갱신하고, 필요 시 모듈별 세부 분석 문서(`docs/analysis/`)로 분리한다.
